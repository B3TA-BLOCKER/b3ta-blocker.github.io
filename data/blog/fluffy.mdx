# CVE-2025–24071 — NTLM Hash Leak via .library-ms File
## Vulnerability Summary:
A specially crafted .library-ms file can be used to silently leak NTLM hashes when a user extracts it from a ZIP archive and views it in Windows Explorer.

This happens without execution, clicks, or warnings — due to how Windows handles Library files and network shares.

## Why It Works:

1. '.library-ms' files are legitimate XML-based files that define "Libraries" (e.g., Documents).

2. These can reference remote paths, like \\\\10.10.16.54\\share.

3. When Windows Explorer renders a .library-ms, it tries to load content from the remote path.

4. To access that UNC path, Windows sends NTLM authentication (username + NTLMv2 challenge-response) automatically.

5. If the attacker controls that UNC path (via Metasploit or Responder), they catch the NTLM hash.

### Exploitation Flow:

1. Attacker creates a malicious .library-ms that points to \\\\attacker-ip\\share.

2. Packs it into a exploit.zip.

3. Victim extracts the zip using Windows Explorer.

4. Windows automatically renders the .library-ms, triggering a connection to the remote SMB path.

5. Windows sends NTLM credentials silently to the attacker.

6. Attacker receives and cracks the hash — or reuses it for access.

### Why ZIP Extraction Is Key:

* If .library-ms is downloaded directly, SmartScreen blocks it.
* If it’s extracted from a zip, Windows treats it as local and auto-renders without warning.
* This bypasses security checks and triggers the NTLM leak.

### Components Involved:

![](/static/images/fluffy/components_involved.png)

### Mitigation:

* Patch (when released)
* Block outbound SMB (ports 445, 139)
* Disable NTLM auth where possible
* Educate users: Don’t open random ZIPs
* Use Smart App Control or attack surface reduction rules

### Real-World Use Case:

* Delivered via phishing (ZIP with .library-ms)
* Uploaded to exposed shares (as you did via SMB)
* Triggers automatically on folder view
* Used by APTs to steal domain hashes for lateral movement

CVE-2025–24071 exploits the way Windows Explorer renders .library-ms files by tricking it into silently leaking NTLM hashes when the file is extracted from a ZIP archive and viewed — no clicks needed.

The `outbound` of Service Accounts has `GenericWrite` to three users, so `p.agila` also has `GenericWrite` access. I now need to abuse this.

The GenericWrite permission in Active Directory `allows a user to modify all writable attributes of an object, except for properties that require special permissions such as resetting passwords.`

If an attacker gains GenericWrite over a user, they can write to the `servicePrincipalNames` attribute and initiate a targeted Kerberoasting attack.
Moreover, `having GenericWrite over a group enables them to add their account` directly to that group, effectively escalating privileges.

Alternatively, if the attacker obtains GenericWrite over a computer object, `they can modify the msds-KeyCredentialLink attribute` to create Shadow Credentials and authenticate using Kerberos PKINIT.

## Add User to Outbound Object

The outbound of Service Accounts has three services: `CA_SVC`, `LDAP_SVC`, `WINRM_SVC`.

I chose to enumerate `WINRM_SVC` because:

- Commonly used for **remote shell**, **PowerShell remoting**, and **WMI**
- Often added to **Remote Management Users** or even has **local admin rights**
- If it is bound to a **SPN** for `WSMAN/` or `HTTP/` you can abuse RBCD to impersonate `Administrator` to **WinRM**

'ESC16 - Security Extension Disabled'

**Attacker** = `p.agila` 
**Victim** = `CA_SVC`

The exploitation mechanisms for ESC16 are identical to those for ESC9 because the end result — a certificate lacking the SID security extension — is the same. The important difference is that for ESC16, any certificate template enabling client authentication and issued by this misconfigured CA can be used in the UPN manipulation attack, not just a template specifically flagged with CT_FLAG_NO_SECURITY_EXTENSION.
